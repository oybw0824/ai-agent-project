# Agent 限流降级组件 — 需求规格说明书（PRD）

**文档版本**：1.0.0 | **状态**：已评审 | **作者**：com.nbcb

---

## 1. 背景与目标

### 1.1 背景

agent-server 已具备细粒度的运行时治理能力（会话超时、模型调用次数、Token 预算、空转检测、工具可用性），但缺少粗粒度的入口开关能力：

| 缺失能力 | 影响 |
|---------|------|
| 按用户/机构控制 Agent 访问 | 新 Agent 上线只能"全开"或"全关"，无法灰度 |
| Agent 全局开关 | 紧急下线 Agent 需要重启服务或修改代码 |
| API → Agent 路由管控 | 无法控制哪些 HTTP 路径映射到哪个 Agent |
| 数据库化动态配置 | 所有治理配置硬编码在 yaml 中，变更需重启 |

### 1.2 目标

1. **双重拦截**：API 层和 Agent 层各设一道防线，互为补充
2. **数据库化**：配置存储在数据库中，本地缓存，变更后自动生效，无需重启
3. **多级权限**：用户级 > 机构级 > 全局，支持时效性白名单实现灰度发布
4. **零代码接入**：独立 Starter 模块，消费方只需引入依赖 + 建表，无需修改业务代码
5. **兜底安全**：内部调用（定时任务/MQ/RPC）不经过 HTTP 入口，也需被拦截覆盖

## 3. 用户故事

| ID | 角色 | 故事 | 验收标准 |
|----|------|------|---------|
| US-01 | 运营人员 | 当新 Agent 上线时，我希望先只开放给白名单用户，验证通过后再全量开放 | ① 可添加用户白名单记录；② 白名单用户在有效期内可正常访问；③ 非白名单用户被拒绝并看到"功能已下线"提示 |
| US-02 | 运营人员 | 当某个 Agent 出现严重故障时，我希望能紧急关闭该 Agent，无需重启服务 | ① 修改全局配置为关闭；② 短时间内所有请求被拒绝；③ 无需重启 agent-server |
| US-03 | 运维人员 | 当某个机构违规使用时，我希望只禁用该机构而不影响其他机构 | ① 添加机构级禁用配置；② 该机构所有用户请求被拒绝；③ 其他机构用户不受影响 |
| US-04 | 开发人员 | 接入治理组件时，我不希望修改现有的 Controller 或 Agent 代码 | ① 引入依赖后自动生效；② 无需在 Controller 上加注解；③ 无需修改 Agent 创建逻辑 |
| US-05 | 开发人员 | 内部定时任务调用 Agent 时，我希望治理仍然生效（兜底保护） | ① 定时任务内部调用 Agent 时拦截仍生效；② 被禁用的 Agent 返回结构化拒绝结果 |
| US-06 | 测试人员 | 灰度白名单应有有效期，过期自动失效，避免遗忘回收 | ① 可设置失效时间；② 过期后白名单用户也被拒绝；③ 可延长有效期 |

## 4. 功能需求

### 4.1 FR-01：单表配置存储

所有治理配置集中存储，通过配置类型字段区分 AGENT（全局）/USER（用户级）/ORG（机构级）/ROUTE（路由映射）四种类型。不同类型对应的必填信息不同（如全局类型必须指定 Agent 名称），需保证同一维度下配置不重复。

### 4.2 FR-02：API 层拦截

**要求**：在 HTTP 请求到达业务接口之前完成校验，业务代码无感知。

**处理逻辑**：

1. 提取用户标识与机构标识；缺失则直接拒绝，不允许绕过
2. 判断请求路径是否命中已配置的路由规则；未命中则不受管控，直接放行
3. 按用户白名单 → 机构白名单 → 全局配置的优先级依次判断是否放行
4. 拒绝时按接口是否为流式返回对应格式的响应

**拦截范围**：覆盖业务接口，排除管理接口和健康检查/监控类接口。

**身份来源**：用户标识、机构标识来自上游认证网关注入的请求头，不由本组件重新认证。

### 4.3 FR-03：Agent 层拦截

**要求**：拦截所有对 Agent 的调用（无论同步还是流式），覆盖不经过 HTTP 入口的调用场景。

| 场景 | API 层能否覆盖 | Agent 层能否覆盖 |
|------|:---:|:---:|
| 外部 HTTP 请求 | 能 | 能（双重保护） |
| 定时任务 | 不能 | 能（唯一防线） |
| MQ 消费 | 不能 | 能 |
| 内部 RPC | 不能 | 能 |

**约束**：Agent 层拦截不具备用户/机构上下文，只校验 Agent 全局开关。

### 4.4 FR-04：本地缓存

**要求**：配置读取走本地缓存，避免每次调用都查库；缓存过期后自动从数据库重新加载最新值；管理端修改配置后应支持主动触发缓存立即失效，不必等待自然过期。

**容错要求**：数据库连接失败时，缓存继续使用上一次的有效值提供服务，不能因为数据库故障导致治理组件本身变成故障点。

### 4.5 FR-05：三级优先级校验

**优先级**：用户白名单 > 机构白名单 > Agent 全局配置 → 均无配置时默认放行。

**设计理由（安全模型反转）**：
- 本组件的定位是"开关"而非"认证"，因此采用默认放行、显式拒绝的模型，而非传统的默认拒绝、显式放行
- 目的是避免因配置遗漏或缺失导致本应正常运行的业务被误拦截中断

### 4.6 FR-06：时效性窗口

用户级、机构级配置需支持生效起止时间：

- 均未设置 → 立即生效，永久有效
- 仅设置生效开始时间 → 到达该时间后生效，永久有效
- 仅设置失效时间 → 立即生效，到期后失效
- 两者都设置 → 仅在该时间窗口内有效

### 4.7 FR-07：流式/非流式分离处理

被拒绝的请求需按接口是否为流式返回不同格式的响应：

| 接口类型 | HTTP 状态码 | 响应格式 |
|---------|-----------|---------|
| 非流式 | 503 | JSON |
| 流式 | 200 | SSE（`data:` 消息） |

**说明**：流式场景约定返回 200 而非 4xx/5xx，因为部分浏览器的流式客户端无法读取非 200 响应的正文，需要客户端从响应体里的成功标识判断真实结果。

### 4.8 FR-08：自动装配

组件需具备条件化自动注册能力：仅在满足相应运行环境（如 Web 环境、Agent 框架存在）时才注册对应拦截器，并提供总开关控制整体是否启用。管理接口和监控健康检查接口需排除在拦截范围外，避免死循环或误拦。

### 4.9 FR-09：独立模块交付

以独立可复用模块形式交付，遵循业界通用的自动装配约定：

- 消费方已有的核心依赖（Web 框架、AOP、ORM、Agent 框架）不重复引入，由消费方工程自身提供
- 组件自身仅引入必要的运行时依赖（本地缓存库）
- 支持标准的自动装配注册方式，接入无需消费方额外配置类

### 4.10 FR-10：总开关

提供一个总开关配置项，默认开启；关闭后 API 层和 Agent 层拦截均不注册，所有请求不受本组件影响，用于紧急降级场景。
